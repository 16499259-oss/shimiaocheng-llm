# ADR-0005: 前端 escapeAttr 反斜杠转义

- 状态（Status）：Accepted（已落地）
- 日期：2026-07-11
- 决策人：高见远（架构师）/ 主理人齐活林

## 背景（Context）

admin 管理面板（`web/admin/app.js`）在拼接用户列表的操作按钮 `onclick` 时，会把用户名、状态等字段通过 `escapeAttr(...)` 注入到 HTML 属性字符串中，例如：

```js
onclick="deleteUser(${u.id}, '${escapeAttr(u.username)}')"
```

早期 `escapeAttr` 只转义了单引号（`'` → `\'`）和双引号（`"` → `&quot;`），**没有转义反斜杠（`\`）**。这留下一个潜在的属性注入隐患：

- 若某个字段值包含反斜杠（例如用户名含 `\`），未转义的反斜杠会「吃掉」紧随其后的转义字符，破坏转义有效性，进而可能闭合属性引号、注入额外的属性或事件，形成 self-XSS（管理员在自己浏览器中触发）。
- 虽然该场景依赖管理员自身输入恶意用户名并被自己查看，属 self-XSS，严重性相对可控，但作为管理后台仍应消除。

## 决策（Decision）

在 `escapeAttr` 中**先对反斜杠做转义**，再转义引号，顺序不可颠倒：

```js
function escapeAttr(str) {
    return str.replace(/\\/g, "\\\\").replace(/'/g, "\\'").replace(/"/g, '&quot;');
}
```

即：反斜杠 `\` → `\\` 优先处理，确保后续的单/双引号转义不会被前置的反斜杠「中和」。修复位于 `web/admin/app.js` 的 `escapeAttr` 函数。

## 后果（Consequences）

**正面：**

- 消除属性字符串中因反斜杠导致的转义失效与潜在 self-XSS 注入面。
- 转义顺序（先反斜杠、后引号）符合 HTML/JS 属性转义的正确做法，逻辑更稳健。
- 改动极小、局部、零副作用，不改变任何业务逻辑。

**负面 / 取舍：**

- 仍属「手动拼接 HTML 字符串」的转义方式，长期看比 innerHTML + 文本节点或框架模板更易出错；本次仅做最小修复，未在 ADR 范围推动架构性改造。
- self-XSS 影响面有限（仅管理员自身），但作为管理后台仍应坚持最小修复闭合隐患。

**后续注意事项：**

- 若 admin 前端改用模板引擎 / 框架，应统一走自动转义，避免再次回到手动escape 的脆弱模式。
- 任何新增的「字符串拼接进 HTML 属性」代码，必须复用 `escapeAttr` 且保持当前转义顺序。
