# PRD: 用户账号有效期管理

| 字段 | 内容 |
|------|------|
| Language | 中文 |
| Programming Language | Go 1.22 + SQLite + Vanilla JS |
| Project Name | `user_expiry_feature` |
| 原始需求 | 管理员在后台创建用户时可设置账号有效期，到期自动返回 403，支持延期 |

---

## 1. 产品目标

1. **自动化停用**：用户到期后 API 自动返回 403，无需管理员手动介入
2. **灵活有效期**：创建时支持 7天/30天/自定义日期/永久四种模式
3. **便捷延期**：管理员可在后台一键延期（+7天/+30天/自定义日期）

**成功标准**：
- 到期用户在 API 请求中 100% 返回 403（无漏网）
- 创建用户+延期操作在 1 次点击 + 1 次确认内完成
- Admin 用户永不受有效期限制

---

## 2. 用户故事

| # | Story |
|---|-------|
| US-1 | As an **管理员**, I want 在创建用户时设置有效期（7天/30天/自定义/永久），so that 可以按需控制用户的访问时长 |
| US-2 | As an **管理员**, I want 在用户列表看到每个用户的有效期和到期状态（过期标红），so that 能快速识别哪些用户需要续期 |
| US-3 | As an **管理员**, I want 点击"延期"按钮给用户增加天数或设置新日期，so that 能在不删除重建用户的情况下延长其使用期限 |

## 3. 需求池

### P0 — 必须实现

| ID | 需求 | 说明 |
|----|------|------|
| P0-1 | `users` 表新增 `expires_at` 字段 | `DATETIME` 类型，`NULL` = 永不过期；通过 `ALTER TABLE` 迁移（idempotent），已有用户默认 NULL |
| P0-2 | Auth 中间件检查到期 | 在 `SubKeyAuth` 中，状态检查后新增：若 `expires_at` 非 NULL 且 `expires_at < NOW()`，返回 403 `{"error":{"message":"API key has expired","type":"key_expired","code":"key_expired"}}` |
| P0-3 | 创建用户时设置有效期 | `POST /admin/api/users` 增加 `expires_at` 字段（可选，不传=永久）；前端创建弹窗增加有效期选择：下拉（7天/30天/自定义/永久）+ 日期选择器（选"自定义"时展示） |
| P0-4 | 用户列表显示有效期列 | `GET /admin/api/users` 返回 `expires_at`；前端列表新增"有效期"列：显示"yyyy-MM-dd"或"永久"；已过期行标红（`expires_at < now` 且非 NULL） |
| P0-5 | 延期功能 | 新增 `POST /admin/api/users/{id}/extend`，接收 `{"days": 30}` 或 `{"until": "2026-01-15T00:00:00Z"}`；前端用户列表新增"延期"按钮 → 弹窗（+7天/+30天/自定义日期） |
| P0-6 | Admin 用户保护 | 创建/延期时若 `role=admin`，忽略 `expires_at`（强写 NULL）；auth 中间件对 admin 角色跳过到期检查 |

### P1 — 应该实现

| ID | 需求 | 说明 |
|----|------|------|
| P1-1 | Dashboard 统计"即将到期用户数" | 新增统计卡片：7天内到期用户数 |
| P1-2 | 审计日志记录延期操作 | `extend` 操作写入 `audit_logs`（action=extend, target_type=user, target_id=userID） |
| P1-3 | 即将到期提醒标记 | 用户列表中，7天内到期的用户行标橙色警告 |

### P2 — 可后续迭代

| ID | 需求 | 说明 |
|----|------|------|
| P2-1 | 到期自动通知 | 到期前 N 天通过 Webhook/邮件通知管理员 |
| P2-2 | 批量延期 | 勾选多个用户后批量 +N 天 |
| P2-3 | 用量到期双控 | "先到先停"：token 用尽 OR 时间到期，任一触发即停用 |

## 5. 待确认问题

| # | 问题 | 建议 | 影响 |
|---|------|------|------|
| Q1 | **已过期用户能否手动"启用"（不改有效期）？** | 建议：手动启用应覆盖到期状态，但下次请求若仍过期则再次 403 | 若选择"启用无效"则需额外提示 |
| Q2 | **延期操作是否需要二次确认？** | 建议：弹窗内点"确认延期"即生效，无需额外 confirm | UI 已设计预览，无需额外确认 |
| Q3 | **过期用户的 quota 数据是否保留？** | 建议：保留，过期仅影响 auth 不删数据，延期后恢复可用 | 当前 quota 无清理逻辑，自然保留 |
| Q4 | **是否需要"到期时间"的 API 文档更新？** | 建议：在 /user/ 自助面板中展示到期时间 | 涉及前端 user 页面改动，为 P2 |
| Q5 | **时间比较用服务器时间还是 DB 时间？** | 建议：用 Go `time.Now()` 服务器时间，与现有 `time.RFC3339` 格式一致 | 需确保服务器时区正确 |
| Q6 | **过期用户的子 Key 是否需要自动清理？** | 建议：无需，过期只是 auth 拦截，子 Key 保留待延期后复用 | 与当前删除行为不同（删除走 CASCADE） |

## 6. 技术实现要点（供架构师参考）

### 6.1 DB 迁移

```
ALTER TABLE users ADD COLUMN expires_at TEXT;
```

通过 `columnExists` 实现 idempotent，已有用户默认 NULL。

### 6.2 中间件修改点

`internal/auth/middleware.go` → `SubKeyAuth()` 方法，在第 65-71 行状态检查之后、Context 注入之前插入：

```go
// Check expiry (skip for admin)
if user.Role != "admin" && user.ExpiresAt != "" {
    expiresAt, _ := time.Parse(time.RFC3339, user.ExpiresAt)
    if time.Now().After(expiresAt) {
        writeAuthError(w, http.StatusForbidden, "API key has expired", "key_expired")
        return
    }
}
```

### 6.3 新增 API 端点

- `POST /admin/api/users/{id}/extend` — 延期
  - Body: `{"days": 30}` 或 `{"until": "2026-01-15T00:00:00Z"}` (mutually exclusive)
  - Response: `{"expires_at": "2026-02-14T...", "message": "..."}`

### 6.4 前端改动文件

- `web/admin/index.html` — 创建弹窗 + 延期弹窗 + 表头新增"有效期"
- `web/admin/app.js` — `createUser()` 传 expires_at、`loadUsers()` 渲染有效期列、`extendUser()` 函数

### 6.5 模型层改动

- `User` struct 新增 `ExpiresAt string \`json:"expires_at"\``
- `CreateUser()` 签名增加 `expiresAt string` 参数
- 新增 `ExtendUserExpiry(db, userID, newExpiresAt)` 函数
- `ListUsers()` SQL 增加 `u.expires_at` 列
- `GetUserBySubKeyHash()` SQL 增加 `u.expires_at` 列
