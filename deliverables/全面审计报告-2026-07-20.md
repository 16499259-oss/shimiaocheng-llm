## 四、问题清单

### P0（阻断）— 0 项

无阻断性问题。鉴权、SQL 注入、并发安全、配额口径均通过审查。

### P1（重要）— 0 项

无重要功能异常或口径错误。

### P2（建议）— 10 项

#### P2-1: 并发 429 日志 ProviderID 硬编码

- **文件**: `internal/proxy/handler.go:263`
- **问题**: 并发超限 429 的 call_log 写入 `ProviderID: "zhipu"` 硬编码，而非实际解析的 provider。此时 provider 尚未解析（并发检查在路由解析之前），所以无法获取真实 provider。
- **影响**: 审计日志中并发拒绝记录的 provider 字段不准确，但不影响功能。
- **建议**: 可将 ProviderID 留空或标记为 `"concurrency_rejected"`，以区分于真实调用记录。

#### P2-2: InsertCallLog 返回值多处被忽略

- **文件**: `internal/proxy/handler.go:261, 468, 509, 530, 558`；`internal/proxy/passthrough.go:204, 239, 267`；`internal/proxy/stream.go:48, 68`
- **问题**: 多处 `models.InsertCallLog(...)` 的返回值被忽略。stream.go:185 是唯一检查 error 的位置。
- **影响**: 调用日志写入失败时静默丢失审计数据，无日志输出。
- **建议**: 统一使用 `if _, err := models.InsertCallLog(...); err != nil { log.Printf(...) }` 模式。

#### P2-3: 审计日志 JSON 拼接可能产生无效 JSON

- **文件**: `internal/admin/users.go:200-206`
- **问题**: 审计日志 detail 使用 `fmt.Sprintf` 拼接 JSON 字符串，若 `req.Username` 或 `req.RouteMode` 包含双引号，会产生无效 JSON。
- **影响**: 审计日志 detail 字段可能无法被 JSON 解析器正确解析。
- **建议**: 使用 `json.Marshal` 构造审计日志 detail。

#### P2-4: 静态文件鉴权绕过检查使用裸字符串前缀

- **文件**: `internal/admin/handler.go:113, 118`
- **问题**: 使用 `r.URL.Path[:8] == "/static/"` 裸字符串切片比较，虽功能正确但不够健壮和可读。
- **影响**: 无安全风险（Go HTTP server 清理路径），但代码可读性差。
- **建议**: 改用 `strings.HasPrefix(r.URL.Path, "/static/")`。

#### P2-5: 用户面板 data.status 未转义

- **文件**: `web/user/index.html:466`
- **问题**: `'🚫 ' + data.status` 直接拼入 innerHTML，未调用 escapeHtml。
- **影响**: `status` 值来自后端（"active"/"disabled"/"deleted"），当前无 XSS 风险，但若后端扩展状态值可能引入 XSS。
- **建议**: 添加 `escapeHtml(data.status)` 或使用 `textContent`。

#### P2-6: 用户面板 expiresAt 在日期无效时未转义

- **文件**: `web/user/index.html:377`
- **问题**: `renderExpiryRow` 中当日期解析失败时，`${expiresAt}` 直接拼入 innerHTML。
- **影响**: `expires_at` 来自后端，当前为 RFC3339 格式，风险极低。
- **建议**: 添加 `escapeHtml(expiresAt)`。

#### P2-7: 用户面板 Sub-Key 存储在 localStorage

- **文件**: `web/user/index.html:307`
- **问题**: Sub-key 存储在 `localStorage` 中以便自动登录。
- **影响**: 若页面存在 XSS 漏洞，Key 可能被窃取。这是常见模式但存在安全隐患。
- **建议**: 考虑使用 sessionStorage 替代，或添加 CSP 头部降低 XSS 风险。

#### P2-8: 路由规则时间字段未转义

- **文件**: `web/admin/app.js:593-594`
- **问题**: `r.start_time` 和 `r.end_time` 在路由规则表格中直接拼入 innerHTML，未调用 `escapeHtml`。
- **影响**: 这些值由 admin 设置且有格式约束，风险极低，但与代码其他部分的一致转义风格不符。
- **建议**: 添加 `escapeHtml(r.start_time)` 和 `escapeHtml(r.end_time)`。

#### P2-9: CreateUser 生成两次 Sub-Key

- **文件**: `internal/admin/users.go:126, 189`
- **问题**: CreateUser 先用 `userID=0` 生成一次 sub-key（仅用于 INSERT），随后用实际 `user.ID` 重新生成并更新。第一次生成的 key 被浪费。
- **影响**: 轻微性能浪费，无功能影响。
- **建议**: 可在 INSERT 时先留空，拿到 user.ID 后再生成；或接受当前设计（保证 key 与 user.ID 关联）。

#### P2-10: stream.go 硬编码 statusCode = 200

- **文件**: `internal/proxy/stream.go:172`
- **问题**: 流式成功路径 `statusCode := http.StatusOK` 硬编码为 200。
- **影响**: 由于 line 56 已过滤 `resp.StatusCode != http.StatusOK`，此处只有 200 响应能到达，实际无 bug。但硬编码不够防御性。
- **建议**: 改为 `statusCode := resp.StatusCode` 以保持一致性。
