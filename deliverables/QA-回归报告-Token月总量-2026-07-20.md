## 五、遗留风险（边界表现）

1. **前端 token 行全隐藏时的观感**：若某用户所有 token 类 limit=0（未设限制）且次数配额也 limit=0，则仅显示「配额无效」行 + 累计/今日 Token。符合需求，但 UI 上 token 区可能「看起来空」——属预期，建议在验收时由产品确认此情形是否需占位提示。
2. **后端一次性重算的触发时点**：重算仅在「marker 列不存在」的首部署触发。若某环境已先行部署过含 marker 列的版本，则不会重算——但此情形本就意味着已重算过，无重复计数风险。
3. **`CAST(x+0.999999 AS INTEGER)` 精度**：double 精度对当前量级（数百万级以下）整数 ceil 精确；零 token 行 `(0)*mult+0.999999 → 0`，安全；倍率为负等异常值不在业务范围内。
4. **`AddTokenUsage` 调用方**：其 `delta` 由调用方按 `ceil((p+c)*multiplier)` 计算（已有 `call_stats_multiplied_test.go` 覆盖），与迁移口径一致；本次未改动调用方，仅确认一致性，无需回归。

**总体评估：改动可发布，无阻塞性问题。**
